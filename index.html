1. Visão Geral da Plataforma

A Eco Express 360° é uma plataforma web integrada de gestão desenvolvida para a Eco Express, projetada para centralizar o acesso a múltiplos sistemas críticos da operação logística: risco, qualidade, frota, auditoria, portais externos e indicadores de desempenho.

Objetivo da plataforma

Unificar em um único local de acesso, com autenticação centralizada, controle granular de permissões e integração com sistemas externos (Power BI, VecFleet, Sem Parar, Geotab, Litroz, Silos, etc).

Público usuário

  • Admin/Master: gestaodecustosff.ee@gmail.com — acesso total a todas as funções.
  • Administradores: usuários com perfil ADM, ADMIN ou ADMINISTRADOR.
  • Usuários comuns: MONITOR, DISPATCHER, SUPERVISOR, COORDENADOR, GERÊNCIA, DIREÇÃO — acesso restrito conforme permissões configuradas.

Principais áreas atendidas

  • 🚛 Frota — Checklists quinzenais e anuais, ofensores PCML, ofensores Scorecard, Torres (offline, scorecard, PCML).
  • 🗂️ Backoffice — Consolidado de Acesso, Material, Treinamentos ECO, Templates Prontos.
  • 🛡️ Auditoria — Pilares, Treinamentos ECOEXPRESS, Sem Parar, Serra, Investigação de Acidente, Relatos Vec Fleet, Multas.
  • 📊 Indicadores de Performance — KPIs BSC, Plano de Ação, Power BI DS/Telemetria/Vou Para Lá/Disponibilidade/Monitoramento, Treinamentos Drivers, Academia Dispatchers.
  • 🌐 Portal — Acesso integrado a 9 sistemas externos (VecFleet, Sem Parar, Geotab, Litroz, etc).

URL de Produção

https://ecoexpress360.web.app

2. Arquitetura Geral

A plataforma utiliza arquitetura baseada em Firebase Hosting com Firebase Authentication, Firestore e Cloud Functions para backend. O frontend é HTML/JS/CSS puro, sem frameworks pesados.

Componentes Principais

ComponenteFunção
Firebase HostingServir os arquivos estáticos da plataforma
Firebase AuthAutenticação de usuários via email/senha
FirestoreBanco de dados NoSQL para perfil, permissões, status
Cloud FunctionsAPI backend para /admin/users, /admin/auth-orphans, etc
Frontend HTML/JSInterface do usuário, sem framework reativo

Fluxo de Funcionamento

  1. Usuário acessa https://ecoexpress360.web.app → carrega public/index.html (shell pai).
  2. Shell pai (index.html) exibe tela de login.
  3. Usuário faz login via Firebase Auth.
  4. Sistema valida o status do usuário no Firestore (users/{uid}).
  5. Se aprovado, exibe header fixo + iframe principal com o dashboard.
  6. Iframe carrega public/dashboard/oficial.html (dashboard filho).
  7. Sistema de permissões do oficial.html aplica regras de data-perm baseado no perfil do usuário no parent.
  8. Botões sem permissão ficam ocultos, grupos vazios somem, abas bloqueadas redirecionam para primeira permitida.

Estrutura de Iframe

A plataforma usa o padrão shell pai + iframe filho:

  • Shell pai (index.html): Header fixo com botões Admin/Sair, barra HOME para portais externos, controle de autenticação.
  • Iframe filho (oficial.html): Dashboard completo com todas as abas, cards, gráficos e botões de menu.

4. Sistema de Login e Autenticação

Como funciona o login

  1. Usuário acessa https://ecoexpress360.web.app.
  2. Tela de login é exibida (card centralizado com logo Eco Express).
  3. Usuário digita e-mail e senha.
  4. Firebase Auth valida credenciais via signInWithEmailAndPassword.
  5. Sistema consulta Firestore: users/{uid}.
  6. Verifica campo status: aprovado, pendente, bloqueado, reprovado.
  7. Se aprovado, monta objeto window.currentUserProfile e window.ecoCurrentUser.
  8. Carrega o iframe com oficial.html.
  9. Sistema de permissões do oficial.html aplica regras baseadas no perfil.

Auto-login

O sistema mantém o usuário logado via Firebase Auth (sessão persistente). Ao recarregar a página, o onAuthStateChanged detecta a sessão ativa e faz login automático se o status for aprovado.

Cadastro

Modal "Criar cadastro" coleta nome, e-mail, WhatsApp, bases (multi-seleção), perfil. Salva no Firestore com status pendente. Aguarda aprovação do admin.

Esqueci minha senha

Modal "Esqueci minha senha" usa sendPasswordResetEmail do Firebase Auth.

Status do usuário

StatusResultado do login
aprovadoLogin permitido. Dashboard carrega.
pendenteLogin bloqueado. Mensagem: "Seu cadastro está pendente de aprovação."
bloqueado / excluidoLogin bloqueado. Mensagem: "Seu acesso não está liberado."
reprovado / rejeitadoLogin bloqueado. Mensagem: "Seu cadastro não foi aprovado."

Admin/Master

  • Admin Master: gestaodecustosff.ee@gmail.com — sempre acesso total.
  • Admin: perfis ADM, ADMIN, ADMINISTRADOR — acesso total, exceto se explicitamente revogado em permissões.

5. Sistema de Permissões

O sistema de permissões é o coração da segurança da plataforma. Ele garante que cada usuário só veja o que está autorizado.

Onde as permissões são salvas

No Firestore, no documento do usuário:

users/{uid}.permissoes.tabs

Exemplo de documento:

{
  uid: "abc123",
  email: "joao@ecoexpress.com.br",
  perfil: "MONITOR",
  status: "aprovado",
  permissoes: {
    tabs: {
      checklist_quinzenal: true,
      ds_power_bi: true,
      litroz: false
    }
  }
}

Regra de Avaliação

  • tabs[chave] === true → mostra o botão/aba.
  • tabs[chave] === false, null, undefined ou ausente → esconde o botão/aba.
  • Admin/Master → vê tudo, independente das permissões.

Como funciona o fluxo de aplicação

  1. Ao carregar oficial.html, o script adiciona classe eco-perm-pending no <html>.
  2. CSS html.eco-perm-pending deixa menu e abas invisíveis (visibility hidden).
  3. Sistema v3 aguarda perfil do usuário estar disponível no parent.
  4. Sistema aplica permissões: esconde botões sem acesso, esconde grupos vazios, redireciona aba ativa para primeira permitida.
  5. Se nenhuma aba liberada: mostra mensagem "Acesso não liberado".
  6. Adiciona classe eco-perm-ready e remove eco-perm-pending — UI fica visível.
  7. CSS html.eco-perm-ready mostra o menu e abas conforme aplicação.
🔒 Sem flash inicial: a regra eco-perm-pending garante que nenhum botão/aba apareça antes das permissões serem aplicadas. O usuário nunca vê conteúdo bloqueado nem por 1 segundo.

Regra do Admin/Master

O email gestaodecustosff.ee@gmail.com e os perfis ADM, ADMIN, ADMINISTRADOR sempre têm acesso total. A verificação é feita em todas as funções do sistema v3.

Componentes do Sistema v3

ComponenteFunção
html.eco-perm-pendingCSS que esconde tudo enquanto permissões não foram aplicadas.
html.eco-perm-readyCSS que mostra tudo depois que permissões foram aplicadas.
data-permAtributo em cada botão que define sua chave de permissão.
ecoOficialGetParentUserProfile()Busca o usuário do shell pai.
ecoOficialUserIsAdmin()Verifica se é admin/master.
ecoOficialPodeAcessarPerm(perm)Verifica se tem acesso a uma permissão.
ecoOficialAplicarPermissoes()Aplica todas as regras (esconde botões, grupos, redireciona).
ecoOficialRedirecionarSeAbaAtualBloqueada()Se aba ativa não é permitida, redireciona para a primeira liberada.
ecoOficialMostrarMensagemSemPermissao()Mostra a mensagem "Acesso não liberado" quando não há nenhuma aba liberada.

Comportamentos do Sistema

  • Bloqueio de aba ativa: se o usuário abrir uma aba via URL ou clique que não tem permissão, o sistema redireciona automaticamente para a primeira aba liberada.
  • Grupos vazios: se um grupo (FROTA, PORTAL, etc) não tem nenhum item liberado, o grupo inteiro some.
  • Mensagem amigável: se nenhuma aba for liberada, mostra a mensagem "Acesso não liberado — Solicite acesso ao administrador".
  • Bloqueio de mudarAba: qualquer chamada de mudarAba() é interceptada e validada antes de abrir a aba.

8. Power BI

Os relatórios do Power BI são a peça central de business intelligence da plataforma. Eles são incorporados via iframe dentro do dashboard.

Como funciona

  1. Usuário clica em um botão Power BI (ex: DS POWER BI).
  2. O botão chama window.ecoAbrirPortalSeguro(url).
  3. Esta função busca o iframe eco-dashboard-frame no shell pai.
  4. Chama window.parent.ecoAbrirPortalExterno(url) no parent.
  5. O parent troca o src do iframe para a URL do Power BI.
  6. Mostra a barra "← Voltar para Plataforma" (ecoPortalBar).
  7. Quando o usuário clica em Voltar, o parent troca o src de volta para /dashboard/oficial.html.

window.ecoAbrirPortalSeguro = function(url) {
  try {
    if (window.parent && typeof window.parent.ecoAbrirPortalExterno === "function") {
      window.parent.ecoAbrirPortalExterno(url);
      return;
    }
  } catch (e) {}
  window.open(url, "_blank", "noopener,noreferrer");
};

15. Regras de Segurança Operacional

⚠️ Regras importantes para manter a plataforma estável:
  1. Nunca mexer no menu/header sem backup — o shell pai controla login, autenticação e header fixo. Qualquer alteração incorreta quebra o acesso.
  2. Nunca alterar Power BI sem testar iframe — os 5 botões Power BI dependem de app.powerbi.com permitir iframe. Se o Power BI mudar essa política, todos os botões param de funcionar.
  3. Nunca alterar permissões sem usuário comum de teste — sempre testar com um usuário que tem apenas 1-2 permissões para garantir que o sistema v3 está escondendo corretamente.
  4. Nunca fazer deploy sem preview — sempre testar localmente ou em preview antes de fazer deploy em produção.
  5. Sempre criar checkpoint antes de mudanças grandes — backup completo de todos os arquivos antes de cada deploy significativo.
  6. Sempre manter DS POWER BI separado do DS interno removido — DS POWER BI é um relatório do Power BI. DS interno (Last Mile DS) foi removido. Não confundir.
  7. Não criar data-perm em conflito — sem_parar (Auditoria) e portal_sem_parar (Portal) são chaves diferentes. Não misturar.
  8. Não adicionar atrasos longos como solução — o sistema de permissões aplica permissões imediatamente. Não usar setTimeout(...4000) ou setTimeout(...6000) para "compensar" o sistema.
  9. Não usar fallback fixo para abas — não usar 'consolidado_acesso' como fallback. A primeira aba permitida deve vir do primeiro botão visível com data-perm e data-aba.

17. Troubleshooting

Problema 1: Botão aparece mas não abre

Causa provável: onclick inline não executa (conflito com dropdown ou listener global).

Onde verificar: public/dashboard/oficial.html, especificamente os botões que não abrem.

Como corrigir: usar o padrão de bind direto via JavaScript, como em bindPowerBIButtons(). Não depender só de onclick inline.

Problema 2: Botão não some mesmo sem permissão

Causa provável: CSS com !important sobrescrevendo o display: none inline.

Onde verificar: estilos CSS que afetam .nav-tab.sub-tab, .sub-nav-group.

Como corrigir: usar btn.style.setProperty('display', 'none', 'important') no JavaScript do oficial.html.

Problema 3: Tela branca

Causa provável: aba ativa sem permissão e nenhuma aba liberada para redirecionar.

Onde verificar: função ecoOficialRedirecionarSeAbaAtualBloqueada.

Como corrigir: garantir que ecoOficialEsconderTodasAsAbas() + ecoOficialMostrarMensagemSemPermissao() sejam chamados quando não há nenhuma aba permitida.

Problema 4: Iframe bloqueado por CSP

Causa provável: site externo retorna Content-Security-Policy: frame-ancestors 'none' ou X-Frame-Options: DENY.

Onde verificar: console do navegador para mensagens de CSP.

Como corrigir: mudar o link para abrir em nova guia usando window.open(url, '_blank', 'noopener,noreferrer'). Não tentar forçar iframe.

Problema 5: Power BI não abre

Causa provável: URL do Power BI mudou, expirou, ou tem bloqueio CSP.

Onde verificar: console do navegador, aba Network para ver a requisição ao Power BI.

Como corrigir: atualizar URL no POWERBI_LINKS dentro de bindPowerBIButtons().

Problema 6: Usuário sem permissão vendo tela

Causa provável: sistema v3 não aplicou permissões ou aba ativa não foi redirecionada.

Onde verificar: console — [ECO-PERM] deve aparecer quando permissões são aplicadas.

Como corrigir: verificar ecoOficialTemPerfilCarregado e ecoOficialRedirecionarSeAbaAtualBloqueada.

Problema 7: Admin/Sair sumiram

Causa provável: shell pai não está exibindo os botões após login.

Onde verificar: public/index.html, função showTopBtns.

Como corrigir: verificar window.__ecoIsAdmin() e os listeners dos botões.

Problema 8: Cache do Firebase

Causa provável: navegador está servindo versão antiga em cache.

Onde verificar: DevTools → Network → Disable cache.

Como corrigir: usar cache-bust na URL: ?v=20260709_permissions_no_flash_final_fix.

Problema 9: JavaScript aparecendo na tela

Causa provável: erro de sintaxe no JavaScript faz o navegador mostrar o código como texto.

Onde verificar: console do navegador, aba Elements.

Como corrigir: revisar o código JavaScript, especialmente aspas, parênteses e aspas escapadas. Usar console.log para debugar.

20. Glossário

TermoDefinição
iframeElemento HTML que incorpora outra página web dentro da página atual. Usado para carregar o dashboard dentro do shell.
shell paiA página index.html que carrega o dashboard em um iframe. Controla autenticação, header e botões globais.
dashboard filhoA página oficial.html que é carregada dentro do iframe. Contém o menu, abas e o sistema de permissões v3.
data-permAtributo HTML em botões que define a chave de permissão do botão. Ex: <button data-perm="checklist_quinzenal">
FirestoreBanco de dados NoSQL do Firebase. Usado para armazenar perfil, permissões e status dos usuários.
Firebase HostingServiço de hospedagem do Firebase. Usado para servir os arquivos estáticos da plataforma.
CSPContent-Security-Policy. Cabeçalho HTTP que define quais recursos podem ser carregados. frame-ancestors controla quem pode embedar a página.
frame-ancestorsDiretiva CSP que controla quem pode embedar a página em iframe. Se o site não permite, o iframe é bloqueado.
Power BI públicoRelatório do Power BI compartilhado publicamente via URL. Pode ser incorporado em iframe.
admin/masterUsuário com permissão total. Identificado pelo e-mail gestaodecustosff.ee@gmail.com ou perfil ADM, ADMIN, ADMINISTRADOR.
cache-bustTécnica de forçar o navegador a recarregar uma versão nova do arquivo, adicionando um parâmetro ?v=... na URL.
deployAto de publicar os arquivos modificados no Firebase Hosting, usando firebase deploy --only hosting:ecoexpress360.
backup/checkpointCópia de segurança dos arquivos antes de mudanças grandes. Permite restaurar a versão anterior em caso de problema.
CSS !importantRegra CSS com prioridade máxima, que sobrescreve outras regras de mesmo seletor.
visibility: hiddenPropriedade CSS que esconde um elemento mas mantém seu espaço na página (diferente de display: none).
MutationObserverAPI JavaScript que permite reagir a mudanças no DOM. Usado para reaplicar permissões quando botões dinâmicos são criados.
Cloud FunctionsBackend serverless do Firebase. Usado para APIs admin (/api/admin/*).
